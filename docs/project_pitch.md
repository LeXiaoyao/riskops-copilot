# RiskOps Copilot Project Pitch

## 1. 120 字中文简历版项目介绍

RiskOps Copilot 是一个贷后策略分析作品集项目，基于合成消费金融贷后数据，搭建指标字典、异常检测、归因分析、经营报告、CLI、策略评估和 ROI 测算链路。项目围绕 M1 D7 回收率下降场景，定位渠道、区域、客群、触达、产能和合规风险信号，用可复现脚本输出 Dashboard 和业务报告，展示贷后策略、指标治理、归因分析和成本收益评估能力。

## 2. 300 字作品集版项目介绍

RiskOps Copilot 是我为风控 / 贷后策略岗位准备的本地化分析工作台 demo。它不接真实客户数据，而是用合成贷后数据模拟一个完整经营场景：M1 D7 回收率下降后，系统先从指标层识别异常，再按渠道、区域、风险分层、触达过程、作业产能和投诉风险做归因，最后输出 Dashboard、Business Report、CLI 摘要、策略评估和 ROI 测算。

这个项目的重点不是做一个漂亮页面，而是把贷后策略人员的工作链路产品化：先定义指标口径，再判断异常，再找 drivers，再形成管理动作，最后评估策略的成本收益。比如项目会把 ECOM 渠道、山东 / 上海区域、D 分和 A 分客群作为回收率下降线索，同时结合 AI 外呼覆盖下降、PTP 履约下降、华东人均案量上升、投诉模板风险等过程证据，避免直接把单一维度说成最终根因。

v0.6.0 进一步加入 Model Lab 和 ROI，把提升 AI 外呼覆盖、补人工产能、调整减免、资源再分配、D 分优先跟进等策略场景做成离线估算。所有结果都明确标注 synthetic data 和 demo estimate，不代表真实上线或真实财务收益。

## 3. 面试官追问 Q&A

### 1. 这是不是接了真实数据？

不是。这个项目没有接真实金融机构数据，也没有真实客户信息。它使用 synthetic data / 合成数据，只用于作品集和面试演示。

我会这样设计，是因为风控项目很敏感。真实贷后数据里可能包含姓名、手机号、证件号、银行卡、地址等 P4 明文字段，这些不应该进入个人作品集、LLM 上下文或公开仓库。所以这个项目重点展示方法论和工程链路，不展示真实客户数据。

### 2. 为什么用合成数据？

主要是两个原因。

第一是合规和隐私。贷后数据天然敏感，我不希望为了 demo 牺牲数据边界。

第二是可复现。合成数据可以稳定模拟 M1 回收率下降、AI 外呼覆盖下降、产能压力、投诉率升高这些业务信号，让面试官不用依赖公司内部环境，也能看到完整分析链路。

我不会把合成数据的结论当真实业务结论。它的价值是证明我知道要看哪些指标、怎么拆原因、怎么控制边界。

### 3. 这个和真实贷后策略工作有什么关系？

关系在于工作方法是相同的。

真实贷后策略不是只看回收率一条线，而是要把结果指标和过程指标串起来。比如回收率下降，要看渠道结构、风险分层、供应商执行、作业线产能、AI 外呼覆盖、人工触达、PTP 履约、减免使用和投诉风险。

这个项目把这些要素做成了可运行链路。虽然数据是合成的，但分析路径贴近真实贷后复盘：先判断异常，再做归因，再给管理动作，再用策略评估和 ROI 判断优先级。

### 4. ROI 是否可信？

ROI 不能当真实财务结果。项目里也明确写了 demo cost assumptions 和 offline demo estimate。

我做 ROI 的目的不是证明某个策略一定赚钱，而是展示策略评估框架：先定义 case base、单位回收价值、动作成本、预估提升，再算 gross benefit、action cost、net benefit、roi_ratio 和 payback period。

真实业务里，这些假设需要用历史实验、Champion / Challenger、分层回溯和财务口径校准。现在这个版本只是把框架搭出来，不夸大成真实预测。

### 5. 有没有机器学习？

当前 v0.6.0 没有训练真实机器学习模型。Model Lab 现在是规则化的离线策略评估，不是模型训练，也不是因果推断。

我这么做是刻意控制范围。对贷后策略项目来说，先把指标口径、数据边界、异常归因和策略假设讲清楚，比直接堆模型更重要。

下一步如果继续增强，可以加入评分卡、XGBoost、SHAP、PSI、分层 Lift、策略回测和 Champion / Challenger，但前提是仍然保留可解释性和合规边界。

### 6. 和普通 Dashboard 有什么区别？

普通 Dashboard 更多是“展示发生了什么”。RiskOps Copilot 更关注“为什么发生、下一步怎么做、动作值不值得做”。

它不只是展示 M1 回收率下降，还会给出 Top drivers、过程证据、管理动作、策略场景和 ROI 估算。比如它会把 ECOM、山东、上海、D 分、A 分作为归因线索，再结合 AI 外呼覆盖、PTP 履约、减免使用、人均案量和投诉模板风险，形成一条贷后策略分析链路。

所以它更像策略分析工作台，而不是单纯 BI 页面。

### 7. 你本人主要做了什么？

我主要做了三类事情。

第一是业务拆解：把贷后经营问题拆成指标、异常、归因、过程证据、策略动作和 ROI，而不是只做页面展示。

第二是工程实现：搭了 Python 项目结构、合成数据链路、指标元数据、warehouse 构建、异常检测、归因报告、Dashboard、CLI、Model Lab 和 ROI 输出。

第三是边界设计：明确 synthetic data、不接真实数据、不触发真实催收动作、不把 P4 明文字段带入分析层或报告层。

### 8. 下一步怎么变得更硬核？

我会优先做三件事。

第一，增强策略实验能力。比如给每个策略场景补更清晰的实验设计、对照组、观察窗口、成功指标和停止条件。

第二，引入可解释模型，但不为了模型而模型。可以从评分卡或 XGBoost 开始，补 AUC、KS、PSI、Lift、SHAP 和分层稳定性评估。

第三，继续补贷后质检和话术合规链路，比如投诉风险模板、文本质检、合规红线、策略建议审批留痕。但这属于后续阶段，不会混进当前 v0.6.0 的演示范围。
