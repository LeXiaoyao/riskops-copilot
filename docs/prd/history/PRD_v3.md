# RiskOps Copilot 产品需求文档（PRD）

## 0. 文档定位

本文档是一份面向展示 Demo、作品集包装、后续开发拆解的产品详细设计方案。

项目暂定名：**RiskOps Copilot：面向消费金融风控与贷后策略的智能分析工作台**。

产品形态不是单纯的聊天机器人，也不是普通 BI，而是一个面向风险、贷后、催收、经营分析场景的垂直智能工作台。它可以读取本地或远程结构化数据、分析指标波动、生成可视化报告、辅助风险建模、进行催收质检、推荐合规话术，并把基础数分、商分、策略分析、质检和运营文档工作尽可能产品化。

第一期优先做贷后方向，覆盖 M1/M1+ 回收率分析、催收过程指标、供应商/线路/产能/话术/合规/ROI 归因、催收录音或文本质检、话术推荐与人工审批、策略报告生成。

# 1. 产品愿景

## 1.1 一句话介绍

**RiskOps Copilot 是一个给消费金融风控与贷后团队使用的 AI 工作台，帮助策略人员、数分、商分、催收管理者和一线催员快速完成数据分析、异常归因、报告生成、策略建议、合规质检和话术辅助。**

## 1.2 产品核心价值

传统风险与贷后团队里，有大量工作是重复性的：拉数、做透视表、看指标、拆维度、写周报、做 PPT、整理 Excel、追踪供应商、检查催收过程、复盘投诉、测算减免 ROI、写策略说明。RiskOps Copilot 的目标是把这些工作沉淀成一套可调用的垂直技能。

核心价值包括：

1. **替代一部分基础数分和商分工作**  
   自动读取数据，生成指标口径、趋势图、异常点、维度拆解、业务结论和报告初稿。

2. **提高贷后策略分析效率**  
   对 M1/M1+ 回收率、接通率、PTP、投诉率、减免 ROI 等指标进行快速归因，定位线路、产能、供应商、话术、客群结构、账龄结构等问题。

3. **提升管理者经营洞察能力**  
   让业务负责人可以用自然语言问：为什么回收率掉了？哪个供应商拖后腿？哪个线路产能不足？减免策略有没有真实增益？投诉上升是不是和某个话术有关？

4. **提升催员的执行质量**  
   给一线催员提供上下文摘要、客户分层、推荐话术、沟通注意事项、合规提醒和发送前审批机制。

5. **提升质检与合规覆盖率**  
   对催收录音、转写文本、短信模板、语音模板进行合规审查，识别高压、威胁、辱骂、误导、第三方骚扰、超频触达等风险。

6. **沉淀策略知识资产**  
   把散落在 Excel、PPT、Word、飞书文档、SQL、脚本和经验里的策略知识，沉淀成可复用的指标字典、报告模板、诊断链路和策略 playbook。

# 2. 产品边界与定位

## 2.1 产品不是做什么

第一阶段不做完整企业级系统，不追求真实接入金融机构生产数据，不做真实自动外呼，不做真实短信批量发送，不直接替代催收员作业。

第一阶段也不追求完整 SaaS 化，不做复杂权限体系、多租户、数据脱敏平台、企业审批流、监管报送系统。

## 2.2 产品应该做什么

产品的长期定位不是一个单点贷后工具，而是一个覆盖**贷前、贷中、贷后全生命周期**的风控智能工作台。

它最终应该服务完整风险经营链路：

1. **贷前**：申请准入、审批策略、评分模型、额度定价、渠道质量、首逾表现、风险收益平衡。
2. **贷中**：额度管理、交易风控、提额降额、风险预警、客户迁徙、行为变化监控。
3. **贷后**：逾期监控、回收率归因、催收过程管理、供应商管理、减免 ROI、投诉合规、催收话术质检。
4. **管理层**：风险收益驾驶舱、资产质量监控、策略动作追踪、经营复盘、资源配置建议。
5. **办公自动化**：自动生成 Excel、PPT、Word、飞书文档、策略报告、产品文档和经营分析材料。

第一期之所以优先做贷后，不是因为产品只做贷后，而是因为贷后数据链路更适合做 Demo：指标直观、归因链路丰富、业务故事强、可视化容易出效果，也最贴合当前经验优势。

## 2.3 分期定位

### 第一期：贷后策略分析与催收 Copilot

目标是用贷后场景打出一个完整样板：

- M1/M1+ 回收率监控
- 催收过程指标分析
- 供应商与线路归因
- 减免 ROI 测算
- 投诉与合规分析
- 催收文本/录音质检
- 合规话术推荐
- 人工审批 Mock
- 可视化经营周报
- PPT/Excel/Word/飞书文档草稿输出

第一期要证明：这个产品不是普通聊天机器人，而是能把风控策略分析链路产品化。

### 第二期：贷前 + 贷中风险经营 Copilot

第二期要把大饼扩到完整风控前中后台，重点加入：

**贷前方向：**

- 申请漏斗分析
- 审批通过率分析
- 拒绝原因分析
- 渠道质量分析
- 首逾/FPD 表现分析
- 额度与定价策略模拟
- 评分模型训练与评估
- AUC/KS/PSI/Lift/Score Band 分析
- 规则命中与误杀分析
- 风险收益 trade-off 测算

**贷中方向：**

- 额度使用率监控
- 提额/降额策略分析
- 交易拒绝率与风险命中分析
- 客户行为变化监控
- 风险等级迁徙
- 预警客户识别
- 干预策略推荐
- 贷中策略效果复盘

第二期要证明：RiskOps Copilot 可以从贷后单点扩展成**风控全生命周期智能分析平台**。

## 2.4 TUI 与可视化的关系

TUI 适合作为命令入口和操作控制台，但不适合承载复杂图表。因此产品建议采用：

- **TUI/CLI：作为任务控制台**  
  用于加载数据、发起分析、查看摘要、选择报告、批准触达、管理项目。

- **HTML Report / Local Web Dashboard：作为可视化展示层**  
  用于展示趋势图、瀑布图、热力图、漏斗图、Vintage 曲线、供应商对比图、归因图、策略地图。

- **Markdown / Word / PPT / Excel / 飞书文档：作为办公输出层**  
  用于沉淀周报、复盘报告、策略方案、会议材料、管理汇报。

一句话：**终端负责发号施令，网页和文档负责展示成果。**

# 3. 目标用户

## 3.1 风控策略人员

典型工作：

- 看贷前、贷中、贷后指标
- 做策略分析和策略迭代
- 解释指标波动
- 评估策略效果
- 设计分层、分案、额度、定价、减免、触达策略
- 写策略复盘报告

核心诉求：

- 快速知道指标为什么变了
- 快速定位问题出在哪个客群、渠道、线路、供应商、账龄、策略动作
- 快速把分析结果变成报告和方案

## 3.2 基础数分 / 商分

典型工作：

- 拉数
- 做报表
- 做数据透视
- 生成图表
- 写经营日报/周报
- 整理 PPT
- 支持业务方临时分析需求

核心诉求：

- 不想重复写 SQL 和 Excel 透视
- 想快速生成“有图、有数、有结论”的分析稿
- 想把半成品数据变成管理层能看的材料

## 3.3 贷后管理者

典型工作：

- 管理回收率、成本、投诉、人效、供应商
- 关注 M1/M1+、委外、自营、AI 外呼、减免、法诉等策略效果
- 看供应商排名、线路产能、回收过程、投诉风险
- 需要向上汇报经营结果

核心诉求：

- 快速看清经营盘面
- 找到回收率波动的真实原因
- 判断供应商是否拖后腿
- 判断是否需要调产能、调分案、调减免、调话术

## 3.4 一线催员 / 催收主管

典型工作：

- 接收案件
- 查看客户上下文
- 进行电话、短信、语音、App push、WhatsApp 等触达
- 记录客户反馈
- 承诺还款跟进
- 避免投诉和违规

核心诉求：

- 快速知道这个客户应该怎么沟通
- 需要推荐话术和异议处理建议
- 不想踩合规红线
- 希望系统帮忙总结客户状态和下一步动作

## 3.5 合规 / 质检人员

典型工作：

- 抽查催收录音
- 检查话术合规
- 处理投诉复盘
- 制定话术规范
- 评估外包催收机构合规风险

核心诉求：

- 提高质检覆盖率
- 快速发现高风险话术
- 把投诉案例和话术规范沉淀下来
- 给供应商和催员反馈改进建议

## 3.6 产品经理 / 经营负责人

典型工作：

- 规划风控产品、数据产品、AI 产品
- 写 PRD、汇报 PPT、项目方案
- 评估 ROI 和产品路线图
- 推进跨部门协作

核心诉求：

- 快速把业务想法转成文档
- 快速生成产品方案、需求说明、会议纪要、项目计划
- 把数据分析结果变成可以讲清楚的管理语言

# 4. 总体功能蓝图

RiskOps Copilot 分为七大能力域。

## 4.1 数据工作台

负责读取、识别、清洗、管理数据。

功能包括：

- 本地 CSV / Excel / Parquet / SQLite 读取
- MySQL / PostgreSQL / DuckDB 连接
- 数据字典识别
- 字段语义识别
- 缺失值、异常值、重复值检查
- 数据血缘与数据源记录
- 半成品分析文件识别
- SQL 生成与解释
- 样例数据生成器
- 数据脱敏与哈希化

## 4.2 风险指标引擎

负责沉淀风控条线的核心指标口径。

覆盖：

- 贷前指标
- 贷中指标
- 贷后指标
- 模型指标
- 运营过程指标
- 合规与投诉指标
- ROI 与成本指标
- 供应商与产能指标

## 4.3 异常检测与归因引擎

负责判断指标有没有异常、异常来自哪里、应该怎么解释。

功能包括：

- 环比 / 同比 / 周同比 / 月同比
- 阈值预警
- 分位数异常
- 移动均值偏离
- 节假日/周末效应识别
- 多维分组拆解
- 结构变化归因
- 过程指标归因
- 策略动作归因
- 供应商 / 线路 / 城市 / 产品 / 渠道归因
- 贡献度排序
- 归因瀑布图
- 归因结论自动生成

## 4.4 可视化与报告引擎

负责把分析结果变成能展示的图、表、文档。

功能包括：

- 趋势图
- 漏斗图
- 瀑布图
- 热力图
- 分布图
- 箱线图
- Vintage 曲线
- Roll Rate 迁徙图
- Lift 图
- KS 曲线
- PSI 分布图
- 供应商排行榜
- 线路作战地图
- 催收过程漏斗
- 投诉风险地图
- ROI 测算卡片
- HTML 报告
- Markdown 报告
- Excel 报表
- PPT 大纲
- Word 报告
- 飞书文档草稿

## 4.5 模型实验室

负责贷前、贷中、贷后风险模型训练、评估、解释。

功能包括：

- 数据集切分
- 特征探索
- 特征稳定性分析
- 逻辑回归评分卡
- LightGBM / XGBoost 模型
- AUC / KS / PSI / Lift / Recall / Precision
- 模型解释 SHAP
- 客群分层
- 策略阈值模拟
- 冠军/挑战者模型对比
- 模型报告生成

## 4.6 催收 Copilot

负责催收作业辅助、话术推荐、录音质检和人工审批。

功能包括：

- 案件上下文摘要
- 客户分层与标签解释
- 推荐下一步动作
- 推荐短信 / 语音 / 电话话术
- 话术合规检查
- 录音转写与质检
- 催收强度评分
- 投诉风险评分
- 异议处理建议
- PTP 承诺跟进提醒
- 触达频次检查
- 人工审批后发送
- 触达留痕

## 4.7 办公文档 Copilot

负责把数据和分析转成办公材料。

功能包括：

- 周报生成
- 月报生成
- 经营复盘报告
- 策略方案
- PRD 产品文档
- 会议纪要
- PPT 大纲
- Excel 分析模板
- Word 正式报告
- 飞书文档草稿
- 本地 Markdown 知识库
- 本地文档搜索

# 5. 风控全条线功能规划

虽然第一期优先做贷后，但产品蓝图需要覆盖完整风控条线。

## 5.1 贷前风控模块

### 5.1.1 贷前经营分析

核心指标：

- 申请量
- 授信申请量
- 授信通过率
- 提交率
- 审批通过率
- 放款转化率
- 拒绝率
- 人审占比
- 自动审批占比
- 渠道通过率
- 产品通过率
- 城市通过率
- 客群通过率
- 额度分布
- 定价分布

典型问题：

- 最近通过率为什么下降？
- 是模型收紧、规则收紧，还是渠道客群变差？
- 哪个渠道贡献了更多高风险用户？
- 哪些拒绝规则命中过高？
- 是否存在误杀？

可视化：

- 申请-授信-放款漏斗
- 渠道通过率矩阵
- 拒绝原因 Pareto 图
- 额度分布图
- 定价分布图
- 规则命中热力图

### 5.1.2 贷前模型分析

核心指标：

- AUC
- KS
- PSI
- Lift
- Bad Rate by Score Band
- Approval Rate by Score Band
- Vintage by Score Band
- 模型分稳定性

典型功能：

- 自动训练评分模型
- 自动输出模型评估报告
- 自动生成评分卡分箱
- 自动判断模型漂移
- 自动解释模型特征贡献
- 自动给出阈值策略建议

### 5.1.3 贷前策略模拟

核心功能：

- 调整审批阈值模拟
- 调整额度阈值模拟
- 调整定价策略模拟
- 规则加严/放松模拟
- 不同渠道策略模拟
- 通过率、坏账率、规模、收益的平衡测算

输出：

- 策略收益表
- 风险收益边界图
- 通过率-坏账率 trade-off 图
- 预估利润变化

## 5.2 贷中风控模块

### 5.2.1 额度与交易监控

核心指标：

- 额度使用率
- 提额申请量

- 哪些用户提额后风险抬升？
- 哪些交易场景逾期表现变差？
- 额度使用率提升是不是风险信号？
- 哪些贷中规则拦截效果好？

可视化：

- 额度使用率分布

- 交易场景风险矩阵

### 5.2.2 贷中预警与客户迁徙

核心功能：

- 预警客户清单

- 预警原因解释
- 干预策略建议

## 8.3 异常检测模块

### 8.3.1 功能描述

### 8.3.2 检测方式

第一期可采用统计规则，不需要复杂机器学习：

- 环比下降超过阈值
- 周同比下降超过阈值
- 低于过去 4 周均值若干标准差
- 连续 N 天下降
- 指标与过程指标背离
- 供应商表现低于整体均值
- 某维度占比快速提升且表现较差

### 8.3.3 输出格式

每个异常输出：

- 异常指标
- 异常时间
- 当前值
- 基准值
- 变化幅度
- 异常等级
- 可能原因
- 建议下钻维度

示例：

> M1 D7 回收率最近 7 天从 18.6% 下降到 15.2%，下降 3.4pct，低于过去 4 周均值 2.1 个标准差，异常等级为高。建议优先下钻供应商、线路、账龄结构和首触达时效。

## 8.4 波动归因模块

### 8.4.1 归因框架

回收率波动不应该只看结果，需要按六层拆解：

1. **资产结构变化**  
   案源是否变差？余额段、账龄、风险等级、渠道、产品是否变化？

2. **客户分层变化**  
   高风险客户占比是否提升？失联客户是否变多？还款意愿是否下降？

3. **策略动作变化**  
   分案规则、减免策略、AI 外呼策略、短信策略、保护策略是否变化？

4. **资源配置变化**  
   供应商、线路、催员产能、人均案量是否变化？

5. **过程执行变化**  
   拨打覆盖、接通、PTP、履约、首触达是否变化？

6. **合规与投诉约束变化**  
   投诉管控、敏感客群保护、禁呼规则是否影响触达？

### 8.4.2 归因方法

第一期可用以下方法：

- 分组对比
- 贡献度排序
- 结构变化拆解
- 过程指标相关性
- Top N 维度下钻
- 基准组对比
- 策略前后对比
- 简单瀑布图归因

### 8.4.3 归因输出

输出应包含：

- 结论摘要
- 主要贡献因素 Top 5
- 次要因素
- 证据图表
- 可验证假设
- 建议动作

示例：

> 本周 M1 D7 回收率下降 3.4pct，主要原因不是单纯案源变差，而是“高余额高风险客群占比提升 + 华东线路产能不足 + 供应商 B 接通率下降 + 减免策略使用率降低”共同导致。其中供应商 B 对整体下降贡献约 0.9pct，华东线路贡献约 0.7pct，高余额客群结构变化贡献约 0.6pct。

RiskOps Copilot 接入本地样例数据后，自动完成：

1. 识别指标异常
2. 拆解回收率下滑原因

5. 检查催收话术和投诉风险
6. 生成经营周报
7. 生成 PPT 汇报大纲
8. 为一线催员推荐合规话术

# 13. 核心 Prompt 设计原则

AI 的系统提示词必须带业务边界。

## 13.1 风险分析 Prompt 原则

- 必须基于数据，不得臆造
- 数据不足要说明
- 先结论后证据
- 从结构、过程、策略、资源、合规、ROI 六个角度分析
- 输出要像策略分析师，不像普通 Chatbot
- 每个建议要有可执行动作

## 13.2 催收 Prompt 原则

- 不生成威胁、恐吓、辱骂、骚扰、误导话术
- 不鼓励绕过监管
- 不鼓励骚扰第三方
- 不鼓励深夜、超频、私人渠道触达
- 所有发送必须人工审批
- 优先事实提醒、方案引导、克制沟通

## 13.3 报告 Prompt 原则

- 管理层版本简洁
- 分析师版本详细
- PPT 版本每页一个结论
- Excel 版本重口径和附表
- 飞书版本重协作和待办

# 14. 合规与安全设计

虽然第一期不做企业化落地，但产品设定必须有合规意识。

## 14.1 数据安全

第一期使用合成数据。若用户导入真实数据，应提示：

- 避免导入明文身份证、手机号、银行卡号
- 使用 user_id_hash 替代真实身份
- 本地处理优先
- 报告中不展示个人敏感信息
- 输出样本明细需要脱敏

## 14.2 自动化决策边界

系统不直接做最终授信、催收处置、法律动作或真实外发。系统只生成建议和草稿，关键动作必须人工确认。

## 14.3 催收合规边界

系统内置规则：

- 禁止威胁恐吓
- 禁止辱骂侮辱

- 禁止暴露债务隐私
- 禁止超频触达
- 禁止禁止时段触达
- 敏感客户需要人工复核
- 投诉客户进入保护策略

## 14.4 审计留痕

即使是 demo，也应该设计日志：

## 15A.7 主数据维表与敏感信息加密设计

为了让数据结构更接近真实公司，第一期合成数据里不能只有事实表，还需要补充主数据维表。维表负责统一客户、借据、案件、产品、渠道、供应商、线路、催员、策略等基础实体，避免后续分析全部靠流水表硬拼。

同时，客户敏感信息必须做分层保护：

1. **最底层业务系统/生长库可以模拟保存明文字段**，例如姓名、手机号、身份证号，但仅用于模拟真实系统，不在报表、数仓应用层、管理驾驶舱中展示。
2. **ODS 层可以保留加密字段和哈希字段**，但默认不展示明文。
3. **DWD/DWS/ADS 数仓层和业务管理侧只使用密文、哈希值、脱敏值和业务主键**。
4. **管理者、策略分析、运营督导、供应商管理、催收质检侧默认只看到 customer_id、customer_id_hash、loan_id、case_id、mobile_masked、id_no_hash 等字段**。
5. **真实项目里应避免把身份证号、手机号、姓名等敏感信息直接进入分析报表或 LLM 上下文**。Demo 中也要通过合成数据和脱敏字段表达这个原则。

### 15A.7.1 核心主键设计

贷后链路中，最核心的业务主键包括：

| 主键 | 中文含义 | 用途 |
|---|---|---|
| customer_id | 客户号 | 客户维度唯一标识，贯穿贷前、贷中、贷后 |
| customer_id_hash | 客户号哈希 | 数仓和管理侧默认展示的客户标识 |
| id_no_hash | 身份证号哈希 | 去重、关联、风控建模，但不展示明文 |
| mobile_hash | 手机号哈希 | 触达关联、去重、黑名单匹配，但不展示明文 |
| mobile_masked | 脱敏手机号 | 仅展示后四位或掩码形式 |
| loan_id | 借据号 | 借据维度主键，用于还款计划、还款明细、借据状态 |
| case_id | 案件编号 | 催收案件维度主键，用于分案、催记、动作、回款归因 |
| plan_id | 还款计划ID | 每期应还计划主键 |
| repay_id | 还款流水ID | 还款明细流水主键 |
| action_id | 催收动作ID | 电话、短信、AI外呼等触达动作主键 |
| note_id | 催记ID | 催员作业记录主键 |
| decision_id | 分案决策ID | 分案引擎决策流水主键 |

主键关系：

```text
customer_id 1:N loan_id
customer_id 1:N case_id
case_id 1:N loan_id，视公司归户规则而定
loan_id 1:N plan_id
plan_id 1:N repay_id
case_id 1:N action_id
case_id 1:N note_id
case_id 1:N decision_id
```

### 15A.7.2 敏感信息字段分层

| 字段 | ODS业务系统层 | 数仓/管理侧 | 说明 |
|---|---|---|---|
| customer_name | 可模拟明文 | 不展示 | 客户姓名，仅底层模拟 |
| id_no | 可模拟明文 | 不展示 | 身份证号，仅底层模拟 |
| id_no_hash | 展示 | 展示 | 身份证号哈希，用于去重和关联 |
| mobile_no | 可模拟明文 | 不展示 | 手机号，仅底层模拟 |
| mobile_hash | 展示 | 展示 | 手机号哈希，用于去重和触达链路关联 |

## 15A.9 数仓表对主键和敏感信息的使用规则

### ODS 层规则

- 可以模拟 `customer_name_encrypted`、`id_no_encrypted`、`mobile_encrypted`、`mobile_masked`、`id_no_hash`、`mobile_hash`。
- 不建议在 ODS CSV 中直接生成明文身份证和手机号。如果为了模拟真实业务系统，可以只在 `raw_secure` 目录中生成，并默认不进入数仓加工。
- ODS 层用于模拟业务系统事实，不用于直接展示。

### DWD 层规则

- 只保留 `customer_id`、`customer_id_hash`、`id_no_hash`、`mobile_hash`、`mobile_masked`。
- 不保留明文姓名、身份证、手机号。
- 明细事实表必须能通过 `customer_id`、`loan_id`、`case_id` 关联。

### DWS 层规则

- 以 `customer_id`、`loan_id`、`case_id`、`vendor_id`、`line_id`、`collector_id` 为核心关联键。
- 只保存用于分析的客户标签、区域、年龄段、风险等级、客群分层。
- 不保存任何明文敏感信息。

### ADS 层规则

- 只输出指标和聚合结果。
- 默认不输出单个客户明细。
- 如果输出样本明细，只能展示 `customer_id_hash`、`case_id`、`loan_id` 和脱敏字段。
- 管理驾驶舱不展示姓名、身份证、完整手机号、详细地址。

## 15A.10 新增 DIM 层建模要求

在项目目录中，应把 DIM 层作为和 ODS/DWD/DWS/ADS 并列的数据层：

```text
synthetic_data/
├── dim/
├── ods/
├── dwd/
├── dws/
└── ads/

schemas/
├── dim.sql
├── ods.sql
├── dwd.sql
├── dws.sql
└── ads.sql
```

元数据中也要补充：

```text
metadata/
├── tables.yaml
├── columns.yaml
├── privacy_policy.yaml
└── key_relationships.yaml
```

其中：

- `privacy_policy.yaml`：记录字段敏感等级、是否允许进入数仓、是否允许进入报告、是否允许进入 LLM 上下文。
- `key_relationships.yaml`：记录 customer_id、loan_id、case_id、plan_id、repay_id、action_id、note_id 等主键之间的关系。

字段敏感等级建议：

| 等级 | 含义 | 示例 |
|---|---|---|
| P0 | 非敏感公开分析字段 | product_code, province, dpd_bucket |
| P1 | 业务内部字段 | customer_id, loan_id, case_id |
| P2 | 脱敏个人字段 | mobile_masked, customer_id_hash |
| P3 | 哈希/密文字段 | id_no_hash, mobile_hash, id_no_encrypted |
| P4 | 明文敏感字段 | id_no, mobile_no, customer_name, address |

规则：P4 字段不得进入 DWD/DWS/ADS，不得进入报告，不得进入 LLM 上下文。

# 20. 当前版本建议结论

当前版本的产品定位要更大一些：

> RiskOps Copilot 不是贷后单点工具，而是面向消费金融风控全生命周期的智能分析工作台。第一期选择贷后，是为了用最容易讲清楚、最容易可视化、最贴合经验优势的场景打样；第二期会扩展到贷前和贷中，形成完整的风险经营 Copilot。

第一期最值得做的是：

**贷后经营分析 + M1 回收率异常归因 + 可视化周报 + 催收文本质检 + 合规话术推荐 + 人工审批 Mock。**

第二期要重点扩展：

**贷前申请漏斗、审批通过率、渠道质量、首逾表现、Vintage、评分模型、AUC/KS/PSI、额度定价策略模拟；贷中额度使用、交易风控、风险迁徙、预警客户识别、提额降额策略和干预建议。**

这样产品的叙事会更完整：

1. **第一层：贷后样板间**  
   用 M1 回收率异常归因做出一个能跑、能看、能讲的 Demo。

2. **第二层：贷前贷中扩展蓝图**  
   证明这个产品不是贷后小工具，而是可以覆盖审批、额度、交易、预警、催收、回款、合规的完整风险经营平台。

3. **第三层：办公自动化与管理驾驶舱**  
   让它不仅能分析数据，还能替代一部分基础数分、商分和产品经理文档工作，自动生成 Excel、PPT、Word、飞书文档和管理层报告。

最终 demo 的目标不是证明系统已经能企业上线，而是证明：

> 这套方向有业务价值、有产品形态、有技术实现路径、有可视化展示、有 AI 差异化，也有从贷后扩展到贷前贷中的完整战略想象空间。

# 21. 补充：组织赋能与管理角色扩展

本产品不能只面向一线策略分析师和催员，还需要覆盖业务督导、运营中台、业务线负责人、区域负责人、策略线负责人和风控决策者。产品的价值不只是“帮分析师做图表”，而是把风险经营链条里的管理动作、督导动作、决策动作都数字化、可追踪、可复盘。

## 21.1 业务督导 / 运营中台

业务督导和运营中台不一定每天写 SQL 或训练模型，但他们需要盯执行、盯过程、盯人效、盯供应商、盯异常闭环。

典型工作：

- 监控各业务线、区域、供应商、线路、班组、催员的过程指标。
- 跟进每日分案、触达、接通、PTP、回款、投诉、质检情况。
- 督促供应商和催员完成作业要求。
- 发现异常后推动策略、运营、质检、供应商共同处理。
- 形成日报、周报、专项复盘和整改追踪。

核心诉求：

- 不只是看到指标结果，还要看到“谁没有执行到位”。
- 快速定位异常发生在哪个区域、线路、供应商、班组、催员。
- 自动生成督导清单、整改事项、跟进状态。
- 对异常事项形成闭环：发现、派单、跟进、复盘、固化。

可用功能：

- 运营督导驾驶舱。
- 供应商/线路/催员红黄绿灯。
- 每日异常工单清单。
- 过程指标未达标提醒。
- 催员作业覆盖率检查。
- 整改跟进表自动生成。
- 周度运营复盘报告。

## 21.2 业务线负责人 / 区域负责人

业务线负责人和区域负责人更关注经营结果、资源配置和跨团队协同。

典型工作：

- 关注整体回收率、成本率、投诉率、利润和资源配置。
- 决定供应商分案、区域产能、预算、人力、减免资源。
- 与风控策略、运营、产品、合规、财务沟通经营结果。
- 参与重大策略调整和资源投放决策。

核心诉求：

- 需要一眼看到业务盘面，而不是看一堆底层报表。
- 需要知道本周、本月最该处理的三个问题是什么。
- 需要判断资源应该投到哪里，哪里要收缩，哪里要加码。
- 需要把分析结果变成管理层能听懂的汇报材料。

可用功能：

- 业务线经营驾驶舱。
- 区域/产品/渠道/供应商经营对比。
- 成本收益分析。
- 资源配置建议。
- 管理层摘要自动生成。
- 月度经营复盘 PPT。
- 关键决策事项清单。

## 21.3 策略线负责人 / 风控决策者

策略线负责人和风控决策者关注的是风险收益平衡、策略方向和机制沉淀。他们不只看某个指标，而是判断整个策略体系是否健康。

典型工作：

- 制定贷前、贷中、贷后的整体风险策略方向。
- 评估审批、额度、定价、分案、减免、催收、法诉等策略效果。
- 决定模型、规则、策略阈值、资源倾斜和风险偏好。
- 向管理层解释风险收益变化。
- 推动策略从人治经验走向系统化和自动化。

核心诉求：

- 看到完整风险链路，而不是单点指标。
- 快速评估策略动作是否有效。
- 判断指标波动是结构问题、策略问题、执行问题还是外部环境问题。
- 把历史策略经验沉淀成可复用 Playbook。
- 支持决策前模拟、决策后追踪和策略复盘。

可用功能：

- 风险收益驾驶舱。
- 策略动作效果追踪。
- 策略前后对比分析。
- 规则/模型/阈值模拟。
- Champion/Challenger 策略对比。
- 风控策略复盘报告。
- 策略 Playbook 自动沉淀。

## 21.4 管理闭环功能补充

为了真正服务管理层和督导层，需要补充以下管理闭环功能：

| 功能 | 说明 |
|---|---|
| 异常自动派单 | 指标异常后自动生成待办，分配给供应商、线路负责人或策略人员。 |
| 整改追踪 | 每个异常事项保留责任人、截止时间、处理动作、复盘结果。 |
| 管理层摘要 | 自动把复杂分析压缩成 3 条核心结论、3 个风险点、3 个建议动作。 |
| 决策事项清单 | 自动识别需要管理层拍板的事项，如是否调整供应商分案、是否放开减免、是否增加产能。 |
| 会议材料生成 | 自动生成经营会、策略会、供应商复盘会的 PPT 大纲和会议纪要模板。 |
| 策略动作追踪 | 每一次策略调整都记录原因、动作、影响范围、上线时间、后续效果。 |

# 22. 补充：真实贷后数据仓库与数据字典设计

第一期虽然使用合成数据，但数据结构必须尽量模拟真实金融机构的贷后数据链路。不能只做几个简单 CSV，否则 Demo 会显得像玩具项目。数据底座应该同时模拟两类表：

1. **催收系统/业务系统生长库表**：接近线上系统的原始业务流水，如还款计划、还款明细、借据状态、案件流转、分案流水、催记、外呼记录、短信记录、减免申请、投诉记录等。
2. **数仓中间表与应用层宽表**：便于业务取数、做表、做分析、做看板，如借据日切片、案件日切片、客户日切片、催收过程宽表、回收率指标汇总表、供应商绩效表、策略归因宽表等。

## 22.1 数据分层设计

建议采用四层结构：

| 层级 | 说明 | 示例 |
|---|---|---|
| ODS 原始层 | 模拟催收系统、生长库、还款系统、贷后策略系统的原始表。字段多、流水细、接近生产系统。 | `ods_repayment_plan`、`ods_collection_action` |
| DWD 明细事实层 | 对 ODS 清洗、标准化、去重、统一口径后形成事实表。 | `dwd_due_plan_detail_di`、`dwd_repayment_detail_di` |
| DWS 主题汇总层 | 围绕借据、案件、客户、催收动作、供应商、线路、策略动作形成宽表和日切片。 | `dws_case_status_snapshot_di`、`dws_collection_process_wide_di` |
| ADS 应用指标层 | 直接服务看板、报告、分析和 AI Agent。 | `ads_postloan_dashboard_di`、`ads_vendor_performance_di` |

## 22.6 到期账户数与账户数量预估

贷后分析不能只看已经逾期的案件，也要预估未来到期账户数、入催规模和产能压力。第一期需要在数据设计中加入还款计划和未来到期计划，支持以下能力：

### 核心问题

- 未来 7/14/30 天有多少账户到期？
- 预计有多少账户会流入 M1？
- 当前催收产能是否足够？
- 哪些产品、渠道、城市未来到期压力最大？
- 是否需要提前调整供应商和线路产能？

### 关键指标

| 指标名 | 说明 |
|---|---|
| due_account_count | 到期账户数 |
| due_loan_count | 到期借据数 |
| due_plan_count | 到期计划数 |
| due_total_amount | 到期应还金额 |
| expected_overdue_count | 预计逾期账户数 |
| expected_overdue_amount | 预计逾期金额 |
| expected_case_inflow_count | 预计入催案件数 |
| expected_m1_case_count | 预计M1案件数 |
| required_collector_count | 预计所需催员人数 |
| capacity_gap | 产能缺口 |

### 预估逻辑

第一期可先用规则和历史比例，不必做复杂预测模型：

- 按产品、渠道、风险等级、C卡分、历史首逾率估算未来入催率。
- 按账龄和余额段估算未来 M1 流入。
- 按人均案件数和人均拨打量估算催员产能需求。
- 按供应商历史接通率和回收率估算资源分配建议。

## 22.7 第一阶段数据交付物

第一阶段 Codex 不应该先做 TUI，也不应该先做复杂 Agent，而应该先把数据底座搭好。数据交付物包括：

1. `metadata/tables.yaml`：所有表的元数据，包含表名、中文名、层级、主题、主键、粒度、用途。
2. `metadata/columns.yaml`：所有字段的数据字典，包含字段名、中文注释、字段类型、是否主键、是否可空、口径说明。
3. `schemas/*.sql`：DuckDB/SQLite 可执行建表 SQL。
4. `synthetic_data/ods/*.csv`：ODS 层合成数据。
5. `synthetic_data/dwd/*.csv`：DWD 层中间表。
6. `synthetic_data/dws/*.csv`：DWS 层主题宽表。
7. `synthetic_data/ads/*.csv`：ADS 层应用指标表。
8. `docs/data_dictionary.md`：人工可读的数据字典。
9. `docs/data_lineage.md`：ODS → DWD → DWS → ADS 的血缘说明。
10. `scripts/generate_synthetic_data.py`：一键生成合成数据。
11. `scripts/build_warehouse.py`：一键从 ODS 加工到 DWD/DWS/ADS。
12. `tests/test_data_quality.py`：基础数据质量测试。
