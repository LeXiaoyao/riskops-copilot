# RiskOps Copilot 5 分钟面试讲稿

## 1. 一句话定位（10 秒）

这是一个贷后风险运营自动化工作台，用合成数据演示从异常检测到策略 ROI 估算的完整链路。

如果用更面试化的话说，它不是一个单纯 Dashboard，而是把贷后策略团队日常复盘的一条链路工程化：先发现 M1 D7 回收率异常，再解释为什么掉，最后估算哪些策略动作值得优先做。

## 2. 背景痛点（30 秒）

我设计这个项目时，模拟的是消费金融贷后里很常见的一个场景：M1 D7 回收率突然下滑。

在真实团队里，这类问题往往不是没有数据，而是数据散在不同地方。策略同学先写 SQL 拉回收率，再用 Excel 拆渠道、区域、客群、供应商和作业线，最后手工拼 PPT 给业务和管理层。这个过程慢，而且很容易因为口径不一致出错。

更麻烦的是，回收率下降通常不是单点原因。可能是某个渠道资产质量变化，也可能是 AI 外呼覆盖下降、人工产能不足、PTP 履约变差，甚至还有投诉和话术合规风险。所以我想做的不是一个“看数工具”，而是一个能把指标、异常、归因和策略评估串起来的 demo。

## 3. 解决方案（60 秒）

这个项目的工程链路分成五步。

第一步是合成数据底座。我用 Python 生成消费金融贷后场景的 synthetic data，再用 DuckDB 和 Parquet 搭一个本地分层数仓，从 ODS、DWD、DWS 到 ADS，模拟案件、账单、触达、PTP、回收、减免、投诉和质检等数据。这样做的目的是在不接触真实客户数据的前提下，把真实贷后分析需要的数据形态复现出来。

第二步是指标字典。项目里有 26 个贷后指标，统一定义在 `metadata/metric_dictionary.yaml`，包括 M1 D7 回收率、AI 外呼覆盖率、有效触达率、PTP 履约率、投诉率、减免 ROI 等。所有指标都从这个 YAML 读取口径，不把业务定义散落在 Python 脚本里。

第三步是异常检测。这里不是用复杂模型，而是用配置化规则引擎。比如环比下降、低于阈值、投诉风险上升、产能压力上升，规则都放在配置里，Python 负责执行和输出结构化异常结果。这样方便解释，也方便策略团队调整阈值。

第四步是归因。系统会围绕渠道、区域、风险分层、作业资源和过程指标做贡献度分解。比如它不会简单说“山东导致下降”，而是会比较各分组的回收率变化和案件占比变化，再结合 AI 外呼覆盖、PTP 履约、人均案量等过程证据，形成更接近真实复盘的 driver 解释。

第五步是策略 ROI 估算。项目里把常见动作做成 strategy scenario，比如提升 AI 外呼覆盖、补人工产能、调整减免、D 分优先跟进、资源再分配。Model Lab 里有一个 logistic baseline，用来演示 leakage-safe 的特征工程和 D7 any-payment response 的 demo 级预测；ROI 模块再基于动作成本、预估提升和 case base 计算 gross benefit、action cost、net benefit、ROI ratio 和 payback。这里我不会把它包装成真实收益预测，它的价值是展示策略评估框架。

## 4. 技术亮点（60 秒）

这个项目我重点想展示四个技术和风控意识。

第一是单一权威源。指标口径集中在 `metric_dictionary.yaml`，表、字段、血缘和隐私分级也在 `metadata/` 里维护。这样报告、异常检测、质量校验和 CLI 读的是同一套定义，避免面试项目里常见的“页面一个口径，脚本一个口径”。

第二是隐私分级。我把字段按 P0 到 P4 做了分级，P4 包括姓名、身份证、手机号、银行卡、地址这类明文敏感字段。项目规则是 P4 只能存在于 raw secure 的模拟边界里，不能进入 DWD、DWS、ADS、报告，也不能进入 LLM context。虽然这是合成数据 demo，但我希望从项目结构上体现真实金融数据边界。

第三是 leakage-safe feature engineering。Model Lab 的目标是 D7 any-payment response，所以训练特征只能使用 snapshot 当日之前可观测的信息。像 D7 回收金额、D7 回收率这类结果字段只能用于评估，不能进训练特征；评分日期也有 `score_date <= snapshot_date` 的 guard，避免 future leakage。

第四是确定性的 briefing 生成器。项目没有让 LLM 直接读数据然后自由发挥，而是先由规则、归因和 ROI 模块生成结构化结果，再用确定性模板输出 briefing。这样好处是可复现、可测试，也能控制“管理建议”不越过真实决策边界。

## 5. 局限性（20 秒）

这个项目也有明确边界。

第一，数据是合成的，所以不能代表真实资产质量和真实催收收益。第二，它不做真实外呼、不发短信、不做真实策略决策。第三，当前 logistic baseline 的 AUC 大约是 0.57，只是 demo 级别，用来展示目标定义、特征边界和评估流程，不是生产模型。

## 6. 延伸方向（30 秒）

如果继续往 Phase 2 做，我会优先接真实数仓的脱敏宽表，让指标字典和质量校验先跑在真实数据口径上。

然后加一层 LLM，但我不会让 LLM 直接做风控决策，而是让它基于已经确定的异常、归因和 ROI 结果生成自然语言经营报告，面向风险总监、策略负责人和作业团队分别输出不同粒度的 briefing。

再往后可以从贷后扩到贷前和贷中，比如贷前通过率、授信质量、贷中额度调整和早期风险预警，但前提仍然是指标口径、隐私边界和可解释性先站稳。

## 7. 常见追问及简短回答

### 1. 你为什么不用真实数据？

因为消费金融贷后数据里天然包含高敏信息，比如姓名、手机号、证件号、银行卡和地址。这个项目是公开作品集，我不应该为了展示能力牺牲隐私边界。所以我用 synthetic data 复现业务形态，重点展示工程链路、指标治理和策略分析方法。

### 2. 这个项目和真实贷后策略工作最接近的地方是什么？

最接近的是分析链路。真实工作里，回收率下滑后也要先确认指标口径，再看异常，再拆渠道、区域、客群、触达、产能、PTP 和投诉风险，最后给动作优先级。这个 demo 把这条链路做成了可运行脚本和可复现输出。

### 3. ROI 估算可信吗？

它不能当真实财务收益。这里的 ROI 是 demo assumptions 下的离线估算，用来说明策略评估框架：先定义目标案件、动作成本、预估提升和回收价值，再计算净收益和回收周期。真实上线前一定要用历史实验、Champion / Challenger、财务口径和风险合规约束校准。

### 4. 为什么 baseline AUC 只有 0.57，还要放进项目？

我刻意保留这个结果，因为它更真实。这个 baseline 的作用不是证明模型很强，而是展示我知道怎么定义目标、避免 leakage、区分训练字段和评估字段，并且不夸大 demo 模型效果。在真实场景里，下一步才是补充更稳定的数据、做分层 Lift、PSI、SHAP 和策略回测。

### 5. 如果你入职后要把它落地，会先做什么？

我会先做三个动作：第一，和风险、策略、数据团队确认指标字典和数据血缘；第二，用脱敏真实数据跑通数据质量、异常检测和归因，不急着上模型；第三，把 briefing 做成管理层和策略同学都能复核的确定性报告。等口径和边界稳定后，再引入 LLM 报告层或更复杂的模型。
