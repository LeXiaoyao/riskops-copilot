# RiskOps Copilot 5-Minute Demo Script

## 1. 开场 30 秒：项目定位

大家好，我先用 5 分钟快速演示一下 RiskOps Copilot。

这个项目不是一个通用 Dashboard，也不是聊天机器人。它更像是一个本地化的贷后策略分析工作台：用合成贷后数据，把指标口径、异常检测、归因分析、经营报告、策略评估和 ROI 测算串成一条完整链路。

我做这个项目的目标，是展示我对贷后策略工作的理解：不是只看一张图，而是从“回收率为什么掉了”一路追到“哪个客群、哪个渠道、哪个过程指标在拖累”，最后再落到“可以试什么策略、成本收益大概怎么看”。

这里先说明边界：项目只使用 synthetic data / 合成数据，不接真实客户数据，不代表真实上线系统，也不会触发真实催收动作。

## 2. 第 1 分钟：业务痛点

贷后策略里有一个很常见的问题：经营指标一波动，团队通常要在 SQL、Excel、PPT、日报和个人经验之间来回切。

比如 M1 D7 回收率下降，管理层会问三个问题：第一，是真的异常还是正常波动？第二，是客群变差、供应商执行变差、产能不足，还是触达策略出了问题？第三，如果要调整 AI 外呼、人工产能、减免策略或供应商资源，哪个动作更值得先试？

RiskOps Copilot 先从结果指标入手，把异常和核心上下文打出来。

演示命令：

```bash
python scripts/riskops_cli.py summary
```

这里我会重点讲几个信息：当前 demo 发现了 8 个异常，其中 high 6 个、medium 2 个；核心问题是 M1 回收率从 19.43% 降到 15.30%，相对下降 21.27%。这不是直接说“业务变差了”，而是把它作为一个需要解释的经营信号。

## 3. 第 2 分钟：异常检测和归因

接下来我会看归因。贷后分析里最容易犯的错误，是看到一个维度下降就直接定责。这个项目刻意把“归因线索”和“最终根因”分开。

演示命令：

```bash
python scripts/riskops_cli.py drivers
```

这里可以看到 Top drivers 指向 channel_code=ECOM、province=山东 / 上海、score_band=D / A。比如 ECOM 这个切片，D7 回收率从 23.90% 降到 6.29%，贡献度是 15.26%。这说明它对整体下降有解释力，但我不会说“ECOM 就是唯一根因”，因为渠道、区域、风险分层之间可能有样本重叠。

我会继续补一句业务判断：D 分客群下降比较符合高风险客群恶化预期，但 A 分客群也下降，说明问题可能不只是坏客群变多，也可能有触达覆盖、作业节奏或资源配置的变化。

过程证据也很关键。比如 AI 外呼覆盖率从 30.09% 降到 17.78%，PTP 履约率从 49.68% 降到 43.58%，华东线路人均案量从 13.54 升到 19.10。这些都不是单独根因，但它们能帮助策略人员形成下一步排查路径。

## 4. 第 3 分钟：Business Report / Dashboard

第三分钟我会切到面向管理者和作品集展示的输出层。

演示命令：

```bash
open outputs/dashboard/dashboard.html
```

Dashboard 的作用不是替代分析，而是让面试官或业务负责人快速看到这个项目的产品形态：它不是一堆散落的脚本，而是可以把异常总览、核心指标、Top drivers、过程证据和管理动作组织到一个可读界面里。

Business Report 则更偏经营复盘。它会把 M1 D7 回收率下降讲成一个业务故事：先说现象，再说 drivers，再说过程证据，最后给管理动作建议。比如优先下钻 ECOM x vendor x line x score_band，复核山东 / 上海区域供应商和作业线，检查 AI 外呼覆盖、PTP 履约、减免使用率和投诉风险模板。

我会强调一个点：报告里所有判断都有边界。它明确写了 synthetic data，不包含真实客户数据，不输出 P4 明文字段，不代表真实业务结论。这个边界对风控项目很重要，因为风控不是只追求“看起来智能”，还要知道什么不能说、什么不能做。

## 5. 第 4 分钟：Model Lab / Strategy Evaluation / ROI

第四分钟我会展示 v0.6.0 新增的 Model Lab。这里的重点不是“我训练了一个模型”，而是先把策略评估框架搭出来。

演示命令：

```bash
python scripts/riskops_cli.py model-lab
python scripts/riskops_cli.py roi
```

Model Lab 里目前有 5 个策略场景：提升 AI 外呼覆盖、补充人工催收产能、调整减免使用策略、供应商与作业线资源再分配、D 分客群优先跟进。

比如提升 AI 外呼覆盖，离线估算里 baseline D7 回收率是 15.2965%，scenario 是 15.8965%，estimated delta 是 0.6000%。这里我会明确说：这是 demo estimate，不是真实预测，也不是因果推断。它的价值是让策略讨论从“我觉得应该加外呼”变成“我们先把假设、影响指标、成本和边界写清楚”。

ROI 部分会进一步把策略动作翻译成成本收益口径。比如提升 AI 外呼覆盖，在 demo 假设下 gross benefit 是 30,000，action cost 是 800，net benefit 是 29,200，roi_ratio 是 36.5。这个数字不能当真实财务结果，但它展示了我会怎么搭建策略评估和 ROI 框架。

## 6. 第 5 分钟：边界、个人贡献、下一步

最后一分钟我会收束到边界和个人贡献。

这个项目目前已经覆盖从数据底座、指标字典、异常检测、归因、Dashboard、Business Report、CLI，到 Model Lab 和 ROI 的完整 demo 链路。它没有接真实数据，没有真实上线，没有自动催收动作，也没有把 ROI 包装成真实收益。

我的个人贡献主要有三块。

第一块是业务建模：我把贷后经营问题拆成结果指标、过程指标、客群结构、产能压力、合规风险和 ROI 几层，而不是只做一个可视化页面。

第二块是工程实现：我用本地 Python 项目把合成数据、指标元数据、warehouse、规则检测、报告输出和 CLI 串起来，保证 demo 可以复现。

第三块是边界控制：项目里反复声明 synthetic data、no real customer data、no real collection action，P4 字段不进入分析层和报告层。

下一步如果继续做，我会优先把它变得更硬核：补更多真实业务形态的策略实验设计、加入模型训练但保留可解释性、增强分层效果评估、补 collection QA 和话术合规链路。不过本轮展示到 v0.6.0 为止，不进入 M7。
